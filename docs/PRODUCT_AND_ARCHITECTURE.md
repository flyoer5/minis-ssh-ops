# SSH AI Agent（Android）正式需求与架构

> 状态：已确认（2026-07-21）  
> 用途：产品定位、技术总纲、版本切分与风险约束。后续实现以本文为准。  
> 原型参考：`ssh-agent-cli/`、`ssh-agent-android/`（Kotlin MVP，非最终架构）

---

## 1. 项目定位

面向**个人运维**场景的 **Android AI SSH 运维工具**。

一句话：

> 在手机上管理自己的 Linux 主机：既能像传统 SSH 客户端一样动手操作，也能通过 **AI Agent** 在安全约束下自动采集、分析并执行运维任务；**数据与密钥仅存本机**；**大模型接口由用户自备**。

### 1.1 非目标（明确不做）

- 不做多人协作 / 团队云托管主机资产
- 不做强制绑定某一家公有模型服务商
- 不做「服务器上必须再装 Agent 守护进程」才能用（以标准 SSH 为主）
- 第一阶段不做 iOS（可后延）

## 2. 四大核心特性

### 2.1 基础 SSH 运维能力

支持连接 Linux 服务器，覆盖个人常用运维操作，长期目标可替代 JuiceSSH 等传统安卓 SSH 工具的主路径能力：

| 能力 | 说明 | 首版 |
|------|------|------|
| 主机连接 | 密码 / 私钥认证，可配端口、用户 | **v0.1 必做** |
| 远程命令 | 单次 / 批量命令下发 | **v0.1 单条**；批量 v0.3 |
| 交互式终端 | 完整 PTY 终端 | **v0.3** |
| 文件传输 | 上传 / 下载（SFTP） | **v0.3** |
| 批量主机 | 多主机列表与批量操作 | 列表 v0.1；批量执行 v0.3 |
| 状态监控 | 基于巡检/指标的主机状态视图 | 巡检 tool **v0.2**；看板可后延 |

### 2.2 AI Agent 智能执行层

区别于纯手动敲命令，接入大模型形成**可约束的自动化运维闭环**：

| 环节 | 说明 | 首版 |
|------|------|------|
| 信息采集 | 硬件、负载、磁盘、服务、日志等（白名单命令） | **v0.2**（可复用现有巡检目录） |
| 逻辑推理 | 按用户目标分析问题、规划分步流程 | **v0.2**（对话 + 计划展示） |
| 约束执行 | 安全规则 + 高危确认后执行部署/排障/配置类动作 | **v0.2 只读 tool + 确认框架**；写操作分级后开 |
| 长效运维 | 定时巡检、有限自愈、批量安装等 | **v0.4** |

**Agent 默认原则：**

1. 默认工具为**只读**；写操作必须分级，高危必须二次确认，极危直接拒绝。  
2. 模型只产出「意图 / 计划 / 拟执行命令」；**真正执行权在本机 Go 引擎与用户确认**，不信任模型原文直接落库执行。  
3. 所有 tool 调用写入本机审计日志。

### 2.3 本地私有化数据存储

以下数据**仅保存在 Android 设备本地**，不上传第三方云端资产库：

- 主机地址、端口、用户名  
- SSH 密码、私钥材料  
- 运维配置、任务与对话历史、审计日志  
- 用户配置的 Base URL / API Key / 模型名  

说明：调用用户**自己配置的**大模型 API 时，请求内容（如日志摘要、命令输出片段）会发往该 API，这是用户显式配置的行为，不是本 App 运营方收集数据。产品文案中需提示用户：勿把极度敏感内容发给不可信的第三方模型服务。

### 2.4 自定义大模型接入

用户可配置（OpenAI 兼容 `chat/completions` 优先）：

| 参数 | 说明 |
|------|------|
| Base URL | 本地网关或第三方兼容服务，例如 `http://<局域网IP>:8317/v1` |
| API Key | 用户自备，仅存本机 |
| 模型名称 | 如 `grok-4.5` 或用户网关支持的其它模型 |

不强制走本产品中转；请求由设备直连用户填写的 Base URL。

## 3. 正式技术架构：Flutter + Go（交付物为 APK）

### 3.1 总览

```text
┌─────────────────────────────────────────────┐
│  Android APK                                │
│  ┌─────────────────┐   HTTP loopback        │
│  │ Flutter UI      │ ◄──────────────────► │
│  │ · 主机管理       │   127.0.0.1:<port>    │
│  │ · AI 对话/任务   │                        │
│  │ · 任务记录       │   ┌──────────────────┐ │
│  │ · 模型配置       │   │ Go 核心服务       │ │
│  └─────────────────┘   │ · SSH 引擎        │ │
│                        │ · Agent 调度      │ │
│                        │ · 加密 SQLite     │ │
│                        │ · 本机 HTTP API   │ │
│                        └────────┬─────────┘ │
└─────────────────────────────────┼───────────┘
                                  │
                    ┌─────────────┴─────────────┐
                    ▼                           ▼
             远程 Linux (SSH)            用户配置的 LLM API
```

| 层 | 技术 | 职责 |
|----|------|------|
| 界面层 | Flutter | 可视化页面、确认弹窗、权限引导；**开发期可用 Chrome Web 调 UI** |
| 核心层 | Go（内置于 APK） | SSH、Agent、加密存储、本机 HTTP |
| 通信 | 本机 HTTP（仅 `127.0.0.1`） | Flutter ↔ Go；**禁止对 0.0.0.0 暴露** |

### 3.2 Flutter

- 开发阶段：优先 `flutter run -d chrome` 调试页面与调用契约（需本机 Go 已启动）。  
- 正式形态：Android 安装包内四大页面——**主机管理、AI 对话（含任务确认）、任务/审计记录、模型与安全配置**。  
- 权限：网络、存储（若需选密钥文件）、后台运行相关引导（电池优化说明见 §7）。

### 3.3 Go 核心

- **SSH 引擎**：连接、认证（密码/私钥）、命令执行、后续 SFTP/会话池。  
- **AI Agent**：组装上下文、调用用户 LLM、解析 tool 计划、安全校验、分步执行、回灌结果。  
- **本地存储**：SQLite + 加密字段（主机密钥类）；应用层 AES（或 SQLCipher，实现阶段二选一并写进 ADR）。  
- **本机 HTTP**：REST（或 JSON-RPC）供 Flutter 调用；仅监听回环地址。

### 3.4 Go 嵌入 Android 的策略

| 方案 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| A. assets 解压 arm64 可执行文件，App 启动后后台拉起 | 迭代快、进程隔离清晰 | 厂商杀后台、SELinux、首次解压与 chmod | **v0.1 可采用** |
| B. gomobile / c-shared `.so` + JNI | 更「原生」、少独立进程 | 集成与调试成本高 | 后台稳定性不够时升级 |

**约束：** 无论 A/B，对外业务端口只绑 `127.0.0.1`。

### 3.5 与现有代码的关系

| 路径 | 角色 |
|------|------|
| `ssh-agent-cli/` | 逻辑原型：巡检白名单、LLM 调用、SSH 连接经验 → **迁入 Go** |
| `ssh-agent-android/` | Kotlin/Compose MVP：真机验证过 UI/巡检/报告 → **功能对照，不作为最终架构** |

新工程建议目录（落地时创建，本文先约定名）：

```text
ssh-ai-agent/                 # 或 monorepo 根
  docs/                       # 本文及后续 ADR
  backend/                    # Go 模块
  app/                        # Flutter 工程
  .github/workflows/          # APK 构建 CI
```

## 4. 开发与出包环境（重要修正）

用户方案原文中「ARM Debian 全流程、无需电脑端构建工具即可稳定出 APK」**不作为承诺**。

| 活动 | ARM Debian 开发机 | 说明 |
|------|-------------------|------|
| Go 业务开发、单测、本机连真 SSH/LLM | ✅ 主战场 | 调试 Agent 最快 |
| Flutter UI + `flutter run -d chrome` | ✅ 主战场 | 解决「反复打 APK 调 UI」 |
| 交叉编译 Android arm64 用 Go 产物 | ✅ 需脚本固化 | 编进 assets 或 `.so` |
| `flutter build apk --release` | ⚠️ **不保证**在 aarch64 Linux 官方 SDK 下始终可用 | 与当前 Kotlin 工程同类限制 |

**正式出包策略（写入流程）：**

1. 日常开发：ARM Debian 上 Go + Flutter Web（或模拟器若可用）。  
2. **Release / 可分发 APK：GitHub Actions（ubuntu x86_64）或任意 x86_64 构建机**。  
3. 真机：adb 安装 CI 产物验证。

文案统一为：

> 业务开发不依赖本机 Android Studio；**APK 由 CI 或 x86_64 环境构建**。

## 5. 版本切分与交付标准

### 5.1 v0.1 — 骨架可装包（建议第一个正式里程碑）

**目标：** 证明 Flutter↔Go↔SSH/LLM 链路，可日常加主机、改模型、跑一条命令。

实现目录：`ssh-ai-agent/`。

- [x] Go：本机 HTTP（health、hosts CRUD、exec 一条命令、settings 读写）  
- [x] Go：SSH 密码/私钥登录；错误可回传 UI（本机已验证）  
- [x] Go：SQLite + 敏感字段 AES-GCM 加密 + `X-Local-Token`  
- [x] Flutter：四页最小可用（主机 / 对话占位 / 记录占位 / 设置）  
- [x] Flutter：对接本机 HTTP；`flutter build web` 已通过  
- [x] Android：启动时拉起 Go（assets + MethodChannel）；仅 loopback  
- [x] CI：`flutter build apk` 产出 `app-debug` artifact（GitHub Actions）  
- [x] 真机：内嵌 Go 启动 + loopback health（adb 安装验证）  
- [ ] 真机：主机配置重启后持久化（用户侧继续验收）  


**非目标：** 完整终端、SFTP、自动 Agent 多步执行。

### 5.2 v0.2 — Agent 闭环

- [ ] 对话页：针对选中主机的多轮上下文

- [ ] 计划展示 + 执行前确认（尤其非只读）  
- [ ] 黑名单拦截（如毁灭性删盘类）  
- [ ] 任务/审计记录可回看  
- [ ] LLM 失败、SSH 断开时的可读错误与重试  

### 5.3 v0.3 — 传统 SSH 补齐

- [ ] 交互式终端  
- [ ] SFTP 上传下载  
- [ ] 批量主机命令  

### 5.4 v0.4 — 长效运维

- [ ] 定时巡检

## 6. 本机 HTTP API 约定（v0.1 草案）

仅监听 `127.0.0.1`。JSON 请求/响应。路径前缀建议 `/v1`。

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/v1/health` | 服务存活 |
| GET/PUT | `/v1/settings/llm` | Base URL / API Key / model（Key 回显可脱敏） |
| GET | `/v1/hosts` | 主机列表（无密钥明文） |
| POST | `/v1/hosts` | 新增主机 |
| PUT/DELETE | `/v1/hosts/{id}` | 更新/删除 |
| POST | `/v1/hosts/{id}/exec` | 执行单条命令（v0.1）；body: `{ "command": "..." }` |
| POST | `/v1/agent/chat` | v0.2：对话与 tool 循环入口 |
| GET | `/v1/tasks` | v0.2：任务/审计列表 |

具体字段在实现时用 OpenAPI 或 `docs/api.md` 固化；**Breaking change 需升 `/v2` 或显式版本协商**。

## 7. 核心安全设计

1. **存储：** SSH 密码/私钥等使用应用层强加密；密钥材料由设备侧密钥体系派生（Android Keystore 参与或用户口令派生），**禁止**硬编码 AES Key。  
2. **Agent：** 内置命令风险分级；极危直接拒绝；高危强制 UI 确认；默认 tool 只读。  
3. **API Key：** 仅存本机；请求直连用户 Base URL，不经本产品中转服务器。  
4. **本机服务：** 只绑定 `127.0.0.1`；可选本地 token，降低同机其它 App 乱调风险（v0.1 起建议启用随机 token，Flutter 启动时注入）。  
5. **后台：** 首次引导关闭电池优化 / 允许自启动（文案说明用途：长任务与 Agent 步骤不中断）。  
6. **架构：** 首版面向 **arm64-v8a** 主流 64 位手机；是否兼容其它 ABI 另议。

## 8. 真机验证清单（每版本发布前）

- [ ] SSH 密码与私钥登录各至少一例  
- [ ] 自定义 Base URL + API Key + 模型可出有效回复（v0.2+）  
- [ ] 杀进程重启后主机与配置仍在，且非明文可见  
- [ ] 本机端口外部网卡不可达（仅 loopback）  
- [ ] 弱网 / 主机不可达 / LLM 4xx·5xx 有明确提示  
- [ ] 高危命令被拦或需确认（v0.2+）

## 9. 方案优势（在修正后的表述）

- 单一 APK 交付，手机直接安装。  
- 资产与密钥本地私有化。  
- 任意 OpenAI 兼容模型 / 本地网关。  
- AI 自动运维与传统 SSH 手动模式并存（分版本达到）。  
- **开发期**用 Flutter Web + 本机 Go 降低 UI 与 Agent 调试成本。  
- **出包**依赖 CI/x86_64，不假装「任意 ARM 开发板官方 SDK 必能 release」。
